- 분석 일자: 2026-07-10
- 분석 범위: Rust 백엔드(`src-tauri/`) 전체, 프론트엔드(`src/`) 전체, 빌드·배포 설정
- 분석 관점: 데이터 안전성 > 보안 > 가용성 > 유지보수성 > 성능 (이 앱의 존재 이유가 "기록을 잃지 않는 것"이므로)

## 1. 치명 (데이터 유실로 직결)

### 1-1. 자동 백업 로테이션이 사실상 무방비였음

**증상.** 백업이 "저장할 때마다" 돌았다. 이 앱의 저장은 체크 하나, 글자 하나 수정에도 발생하므로, 20개 백업 슬롯 전체가 **한 편집 세션 몇 분 안에 전부 소진**된다. 실제 운영 데이터로 확인: 보존된 백업 7개 전부가 같은 30분 구간에 몰려 있었다.

**왜 치명인가.** 백업의 목적은 "잘못된 상태를 알아챘을 때 되돌아갈 지점"이다. 잘못된 상태가 저장 20번을 살아남는 데 몇 분이면 충분하다면, 알아챘을 때는 이미 모든 백업이 오염된 뒤다. 백업이 존재하지만 보호하지 못하는, 최악의 형태(있다고 믿게 만드는)의 무방비.

**수정.**
- 저장 후 로테이션에 **30분 스로틀** 도입 (`rotate_backup_throttled`) — 최신 백업이 30분 미만이면 건너뜀.
- 정리(prune) 정책에 **일자별 보호** 추가: 최근 14개 날짜 각각의 "그날 첫 백업"은 개수 상한과 무관하게 보존. 하루 종일 편집해 최근 슬롯이 전부 갈려도, 날짜별 복원 지점이 2주치 남는다.
- 수동 복원·가져오기 직전의 강제 스냅샷(`preimport_`/`prerestore_`)은 스로틀을 우회(무조건 기록).

**검증.** `throttled_rotation_skips_when_recent_backup_exists`, `pruning_protects_first_backup_of_each_day` 테스트 추가, 통과.

### 1-2. 저장 위치 변경 시 편집 내용이 조용히 유실되는 창(window)이 있었음

**증상.** [저장 위치] 변경은 "선택 시점"에 DB를 새 폴더로 복사했다. 사용자가 "지금 재시작"을 거절하고 계속 일하면, 이후의 모든 편집은 **구 위치에만** 저장된다. 다음 실행 시 앱은 새 위치(선택 시점의 낡은 스냅샷)를 열고 — 그 사이의 편집은 흔적도 안내도 없이 사라진다.

**왜 치명인가.** 유실이 조용하다. 오류도, 경고도 없이 "며칠 전 상태로 돌아간" 앱을 만나게 되며, 사용자는 원인을 추정할 단서조차 없다. 실제로 이번 분석 전 세션에서 이 경로로 데이터가 두 갈래로 갈라진 사고가 이미 발생했었다.

**수정.** 복사 시점을 "선택 시"에서 **"다음 실행의 맨 앞"**으로 이동. `choose_data_dir`은 config에 `data_dir`(새 위치)과 `pending_move_from`(현 위치)만 기록하고, 다음 실행이 DB를 열기 전에 `apply_pending_move`가 **그 시점의 최신 데이터**를 통째로 옮긴다. 따라서 재시작을 미루고 한 편집도 구조적으로 전부 따라간다. 이동 실패(대상 드라이브 소멸 등) 시에는 설정을 구 위치로 되돌려 **실데이터 위에서 계속 동작**한다 — 빈 새 위치로 시작하는 것보다 항상 안전하다.

**검증.** `apply_pending_move_carries_db_backups_and_staged_import` 테스트 추가(대기 중 가져오기 파일·백업 동반 이동, 원본 보존까지 확인), 통과. 같은 원리로 이미 고쳐진 DB 가져오기 staging(`apply_pending_import`)과 실행 순서를 통일(이동 → 가져오기 → 열기).

### 1-3. 다중 인스턴스 실행이 무방비였음

**증상.** exe를 두 번 실행하면 두 창이 모두 뜨고, 둘 다 같은 SQLite 파일에 저장한다. 이 앱의 저장은 **전체 교체**(delete + reinsert)이므로 행 단위 충돌이 아니라 — 창 A에서 항목을 추가한 뒤 창 B에서 아무거나 수정하면 B의 메모리 상태(A의 추가가 없는)가 **데이터셋 전체를 덮어쓴다**. 이번 세션 중 실제로 2개 인스턴스가 동시에 떠 있는 것이 관찰되었다.

**수정.** `tauri-plugin-single-instance` 도입 (플러그인 등록 순서 규약대로 최우선 등록). 두 번째 실행은 프로세스를 만들지 않고 기존 창을 앞으로 가져온다.

**검증.** 실기동 테스트: 두 번째 실행 프로세스가 즉시 종료되고(exited: True) 프로세스 수 1 유지 확인.

## 2. 높음 (가용성·보안)

### 2-1. 내부망 전용 앱이 Google Fonts를 원격 로딩

`index.html`이 `fonts.googleapis.com`에서 웹폰트를 받아왔다. 이 앱의 배포 대상은 **인터넷이 차단된 내부망**이다 — SheetJS는 바로 그 이유로 파일에 인라인해 두고서, 폰트는 인터넷에 의존하는 자기모순. 내부망에서는 렌더링 지연(차단 응답 대기) 후 어차피 폴백 폰트로 뜬다. **수정:** 원격 링크 전면 제거, Windows 기본 탑재 폰트 스택(`Malgun Gothic` / `Cascadia Mono`·`Consolas`)으로 교체. 외부 네트워크 의존성 0 달성.

### 2-2. CSP 미설정 (`"csp": null`)

이 앱의 렌더링은 전부 innerHTML 문자열 조립이고, 사용자 입력은 `esc()`/`escAttr()` 수동 이스케이프에 의존한다. 지금까지의 호출부는 점검 결과 누락이 없었지만, **단 한 곳의 미래 실수**가 곧 스크립트 주입이고, `withGlobalTauri` 환경에서 주입된 스크립트는 `invoke()` 전권 — 즉 DB 전체 읽기/파괴 권한을 갖는다. 방어선이 "영원히 실수하지 않기" 하나뿐인 구조. **수정:** `script-src 'self'` 기반의 CSP 적용 (인라인 스크립트 실행 원천 차단, 외부 리소스 차단). 2-1과 맞물려 `font-src`/`connect-src`도 자체 리소스와 Tauri IPC로 한정. **검증:** CSP 적용 빌드에서 앱 구동·버튼 핸들러·네이티브 대화상자 호출 정상 동작 확인.

### 2-3. 초기 로드 실패 시 "무언의 죽은 앱"

초기화가 `(async()=>{...})()` 즉시실행에 오류 처리 없이 들어 있었다. `load_all`이 실패하면: 화면은 빈 보드, 저장은 F1 게이트로 차단, 사용자에게는 **아무 안내도 없음**. 사용자 입장에서 "데이터가 전부 사라진" 것과 구별 불가능한 화면이다. **수정:** try/catch로 감싸 실패 시 (a) 저장 차단은 유지하되 (b) 무슨 일이 났는지, (c) 복구 경로([JSON·DB파일 불러오기])가 무엇인지 명시하는 안내를 즉시 표시.

## 3. 중간 (정합성·자원)

### 3-1. JSON 복원이 비원자적 + 원자적 복원 커맨드는 죽은 코드

JSON 불러오기가 items는 `save_all`로, fields/presets/idKinds/settings는 각각 **별도의 fire-and-forget 호출**로 흩어 저장했다. 중간에 앱이 종료되면 반쪽 복원이 남는다. 정작 5개 테이블을 한 트랜잭션으로 복원하는 `backup_import` 커맨드가 백엔드에 이미 있었는데 **프론트가 호출하지 않는 죽은 코드**였다. **수정:** 프론트가 구버전 마이그레이션·누락 섹션 보정을 마친 완전한 payload를 만들어 `backup_import` 한 번으로 복원. DB 반영 성공이 확인된 뒤에만 메모리 상태를 교체(실패 시 "데이터는 변경되지 않았습니다" 보장). 부수 수정: `BackupPayload` 전 필드에 serde 기본값 부여(구버전 백업의 누락 섹션이 역직렬화 전체를 거부하지 않도록), 복원 성공 시 `integrity_ok` 재설정(시작 시 손상 판정 후 복원해도 재시작 없이 정상 동작).

### 3-2. 알람 효과음의 AudioContext 누수

`beep()`이 알람마다 `new AudioContext()`를 만들고 닫지 않았다. 브라우저 엔진은 동시 AudioContext를 약 6개로 제한하므로, 알람이 몇 번 울리고 나면 **효과음이 조용히 죽는다** — 하필 "놓치지 않기"가 존재 이유인 앱에서. **수정:** 싱글턴 재사용 + suspended 상태 resume.

### 3-3. 설정성 테이블의 중복 키가 저장 전체를 폭파

`fields`/`presets`/`id_kinds`는 PK 충돌 시 트랜잭션 전체가 실패한다 — 중복 행 하나 때문에 그 저장의 **모든 변경**이 사라진다(레거시 앱은 중복을 그냥 허용했으므로, 오래된 백업엔 중복이 실존할 수 있다). **수정:** 이 세 테이블만 `INSERT OR IGNORE`(선착순 유지)로 완화. 프리셋은 중복 무시 시 세부항목이 앞 프리셋에 잘못 붙지 않도록 함께 건너뜀. `items`는 의도적으로 엄격 유지 — 항목 id 중복은 진짜 불변식 위반이므로 크게 실패하는 것이 맞다.

## 4. 수정하지 않고 기록만 남기는 항목 (사유 포함)

| # | 항목 | 사유 |
|---|---|---|
| 1 | 저장이 전량 교체(delete+reinsert) 방식 | 수백~수천 건 규모에서 단일 로컬 트랜잭션 수 ms. 차등 upsert는 복잡도만 사고 얻는 게 없음. **1만 건 이상으로 커지면 재검토.** |
| 2 | XLSX 바이트를 JSON 배열로 IPC 전달 | 수십 KB 파일이 수백 KB JSON이 되는 비효율. 현재 규모에서 체감 불가. Tauri raw payload로 바꿀 수 있으나 지금은 YAGNI. |
| 3 | `Notification` API (웹 표준) 사용 | WebView2에서 권한 동작이 불안정할 수 있으나 try/catch로 감싸져 있고, 실제 알림 수단은 모달+강제 포커스+효과음이 담당. 실사용 문제 보고 시 tauri-plugin-notification으로 교체. |
| 4 | `esc()`가 작은따옴표 미이스케이프 | 이 코드베이스의 속성은 전부 큰따옴표로 감싸며 `escAttr()`이 커버. CSP 도입으로 심층 방어 확보. 신규 코드는 속성에 반드시 `escAttr()` 사용할 것. |
| 5 | Mutex poisoning 시 복구 불가 | 락 보유 중 패닉이 나면 이후 모든 커맨드가 실패. 단, 그 시나리오 자체가 버그이며 재시작으로 해소. `parking_lot` 도입은 과잉. |
| 6 | 격리 파일(`wmhh.broken-*`) 무기한 누적 | 의도적: 손상 증거물은 사용자가 직접 지우기 전까지 보존하는 게 맞다. 용량이 문제되면 그때 보존 개수 상한 추가. |
| 7 | 프론트 업무 규칙(placeOf 등) 자동 테스트 부재 | 현재는 레거시와의 동작 일치가 검증 기준. 순수 로직을 모듈로 분리해 테스트를 붙이는 건 가치 있는 후속 작업 (§6). |

## 5. 이번 패스 이전에 이미 해결된 구조 문제 (맥락 기록)

1. **Tauri 웹뷰에서 브라우저식 다운로드(`<a download>`+blob)가 무음 실패** → 네이티브 저장 대화상자(`save_text_file`/`save_binary_file`)로 전면 교체.
2. **SheetJS `{type:'array'}`가 ArrayBuffer를 반환해 `Array.from()`이 빈 배열 생성** → 0바이트 XLSX의 원인. `new Uint8Array()` 래핑으로 수정.
3. **실행 중인 자기 DB 파일 위에 가져오기 파일을 덮어써 재기동마다 손상** → staging 후 다음 실행 벽두에 적용(`apply_pending_import`)으로 수정. 1-2의 수정은 이 원리를 저장 위치 이동에도 일반화한 것.
4. **시작 시 DB를 못 열면 패닉으로 실행 불능** → 3단 복구(원본 → 최신 백업 → 격리 후 새 시작 → 기본 위치 폴백)로 수정. 어떤 저장소 상태에서도 실행은 보장.
5. **"24:00" 입력 시 날짜 이월 누락으로 24시간 오차** (F13), **마감 알람 점이 태그 배경에 묻힘**, **프리셋 세부할일 드래그 정렬 미구현** 등 UI/입력 버그 일괄 수정.

## 6. 권고 (우선순위순)

1. **버전 규율**: 이번에 매니페스트 3종(Cargo/tauri.conf/package.json)을 2.2.0으로 동기화하고 UI에 버전을 표시했다. 이후 릴리즈마다 세 파일을 함께 올릴 것 — 사용자 문의 시 "화면 우상단 버전"이 유일한 식별자다.
2. **업무 규칙의 테스트 가능화**: `placeOf`/`combineDT`/정렬 로직을 `src/logic.js`로 분리하면 Node로 직접 단위 테스트 가능. 자동 배치 규칙이 이 앱의 심장인데 현재 회귀 방어가 수동 확인뿐이다.
3. **배포 전 실환경 점검**: 실제 내부망 PC에서 (a) WebView2 유무, (b) 백신의 exe 차단 여부, (c) `%LOCALAPPDATA%` 쓰기 정책을 1회 확인. 특히 미서명 바이너리 차단(AppLocker) 환경이면 코드 서명이 선행 과제가 된다.
4. **XLSX 필드 확장 여지**: 현재 내보내기는 요약 문자열 위주. 내부망 타 시스템 연동이 구체화되면 기계가 읽기 좋은 별도 시트(정규화된 세부할일 행) 추가를 검토.

## 7. 부록 — "wmhh.sqlite 오류 재발" 신고 조사 (2026-07-10)

보고서 작성 중 사용자로부터 "DB 파일이 또 안 열린다"는 신고 접수. 전수 무결성 검사로 조사한 결과:

- **현재 사용 중인 DB(찐막테스트 위치)는 정상** — 무결성 ok, 신고 직전까지 저장 정상 동작.
- 안 열린 파일 2개(`바탕화면\뭐해야했더라_데이터`의 23:49본, `테스트 폴더\data`의 22:54본)는 **§5-3의 옛 버그가 살아있던 시간대(전날 22:47~23:49)에 이미 손상된 잔해 파일**. 버그 수정은 새로운 손상을 막을 뿐, 그때 망가진 파일을 소급해 고치지 못한다.
- 즉 재발이 아니라, **새 가져오기 검증이 손상 파일을 앞단에서 올바르게 거부**한 것. 다만 거부 메시지가 영어 원문(`file is not a database`)이라 "그 버그가 또 났다"로 오독하게 만들었다 — 이것은 실제 UX 결함으로 인정하고 수정: 거부 사유·"현재 데이터는 변경되지 않았음"·다음 행동(다른 백업 선택)을 한국어로 명시.
- 동반 수정: 가져오기 직전 자동 스냅샷의 파일명이 unix epoch(`wmhh_preimport_1783628958`)로 남아 다른 백업과 형식이 어긋나고 일자별 보호의 날짜 파싱을 오염시키던 문제 → 전 백업 파일명을 `YYYYMMDD_HHMMSS` 단일 형식으로 통일.
- 손상 잔해 2개는 사용자 파일이므로 임의 삭제하지 않음. 정리하려면: `바탕화면\뭐해야했더라_데이터` 폴더와 `테스트 폴더\data\wmhh.sqlite`는 삭제해도 안전하다 (현재 데이터와 무관).

### 손상은 애초에 왜 발생했는가 — 기술적 원인 규명

**한 줄 요약: 실행 중인 앱이 "자기 자신이 열어둔 DB 파일" 위에 다른 파일을 그대로 덮어썼기 때문이다.**

1. **무엇을 했나.** 구버전 '불러오기'는 사용자가 고른 `.sqlite` 파일을 `fs::copy`로 현재 사용 중인 `wmhh.sqlite` 위에 즉시 덮어썼다. 그런데 그 순간에도 앱 프로세스 안에는 그 파일을 열어둔 SQLite 연결이 살아 있었다.
2. **무엇을 어겼나.** SQLite의 절대 규칙: DB 파일의 내용 변경은 반드시 SQLite 자신의 잠금 프로토콜을 거쳐야 한다. 열린 연결 밖에서 파일 바이트를 바꾸는 것은 공식 문서 *"How To Corrupt An SQLite Database File"* 에 이름까지 올라 있는 대표적 손상 시나리오다 (연결은 이미 읽어둔 페이지 캐시·저널 상태·변경 카운터를 "옛 파일 기준"으로 쥐고 있다).
3. **왜 그 자리에서 안 터지고 다음 실행에서 터졌나.** 바이트 복사 자체는 성공한다 (Windows에서 SQLite는 공유 모드로 파일을 열므로 복사가 거부되지 않음). 손상은 그 뒤에 온다 — 이 앱은 수시로 자동저장(save_all)을 하므로, 낡은 상태를 쥔 연결이 **옛 파일 기준의 페이지를 새 내용 위에 흘려 쓰면서** 파일 내부 구조(첫 16바이트 매직 헤더 포함)가 깨진다. 그 세션 안에서는 멀쩡해 보이다가, 다음 실행이 파일을 처음부터 읽는 순간 `SQLITE_NOTADB (extended 26, "file is not a database")`로 드러난다.
4. **증거 3종.** (a) 그 시간대(07-09 22:47~23:49)의 가져오기 시도마다 직전 스냅샷(`preimport_*`)이 남아 있고, 매번 다음 실행에서 "백업으로 복구" 대화상자가 떴다. (b) **완전히 동일한 바이트 복사**를 "연결이 0개인 다음 실행 최전방"으로 옮긴 뒤(staging 방식)로는 같은 작업이 항상 성공한다 — 현재 사용 중인 정상 DB가 바로 그 경로로 들어온 것이다. 즉 변수는 복사 방법이 아니라 **열린 연결의 유무**뿐. (c) 손상 파일 2구의 수정 시각(22:54, 23:49)이 정확히 구버전 코드가 살아 있던 창 안에 있다.
5. **재발 방지 계층.** ① 가져오기·저장 위치 이동 모두 "다음 실행, 연결 0개 시점"에만 파일을 교체 ② 교체 전 자동 스냅샷 ③ 가져올 파일 사전 무결성 검증(손상 파일은 입구에서 거부) ④ 그래도 뚫리면 시작 시 3단 자동 복구. 같은 원인으로는 다시 손상될 수 없는 구조다.

## 9. 후속 데이터 안전 강화 (v2.5.11 ~ v2.5.20 추가 이력)

본 보고서(v2.2.0) 이후 같은 관점("기록을 잃지 않는 것")에서 이어진 데이터 안전 작업 요약. 상세 변경은 `CHANGELOG.md`, 코드 위치·불변식은 `CLAUDE.md` 참조.

- **v2.5.11 — 백업 정렬 유실 버그.** 백업 파일명이 `wmhh_`·`wmhh_prerestore_`·`wmhh_preimport_`로 접두사가 섞이면서, 파일명 문자순 정렬이 `'p'(0x70) > '2'(0x32)` 때문에 오래된 `prerestore` 스냅샷을 "최신"으로 오인 → 복구가 옛 데이터를 되살리고 진짜 최신 백업을 정리(prune)로 삭제했다. **수정:** 파일명이 아니라 파일명에 박힌 `YYYYMMDD_HHMMSS` 스탬프(`stamp_key`)로 정렬. 회귀 테스트로 고정.

- **v2.5.19 — 조용한 저장 실패의 가시화.** 이 앱은 편집마다 백그라운드로 전체 저장(save_all)을 하는데, 그 저장이 실패해도(디스크 꽉 참·저장 위치 분리·백신 잠금·무결성 게이트) `console.warn` 한 줄뿐 화면 신호가 없어, 저장 안 된 채 앱을 닫으면 유실될 수 있었다(v2.31에서 옛 `saveStatus` 표시기를 제거한 뒤 생긴 공백). **수정:** `#saveAlert` 배너를 도입해 실패 시 표시, 다음 *항목* 저장 성공 시 자동 해제("조용한 성공/시끄러운 실패"). 해제는 `save_all` 성공만 담당해 사이드카 저장 성공이 항목 저장 실패를 가리지 않는다.

- **v2.5.20 — 파괴적 저장 3종 하드닝 + 입력 상한.**
  1. **대량 축소 전 강제 스냅샷**(`save_items_guarded`): 전체 대체 저장 특성상 프론트 오류나 실수로 항목이 통째로 줄어든 채 저장되면 전부가 대체된다. 기존 5건 이상 → 절반 이하로 급감하면 30분 스로틀과 무관하게 덮어쓰기 전 `wmhh_presave_*` 스냅샷을 남긴다(F1 게이트는 시작 창만 보호하므로 그 위의 심층 방어).
  2. **복구 시 백업 무결성 재검증**(`open_with_recovery`): 최신순으로 `PRAGMA integrity_check`를 통과하는 첫 정상 백업만 채택 — 손상된 최신 백업을 맹목적으로 되살리지 않는다(수동 `.sqlite` 가져오기 경로와 대칭).
  3. **저장 위치 이동 대상 덮어쓰기 방지**(`choose_data_dir`): 이미 `data/wmhh.sqlite`가 있는 폴더로의 이동은 스테이징 전에 거부 — 다음 실행의 `apply_pending_move` 복사가 남의 데이터를 덮지 않도록.
  4. **입력 길이 상한**: 무제한이던 텍스트에 `maxlength`(메모 50,000·세부 500·연락처/식별정보 100·전화 40·경로 1,000). 비정상 붙여넣기가 매 저장의 전체 직렬화를 부풀리는 것을 막는다. `maxlength`는 새 입력만 제한하고 기존 저장값은 왕복 시 보존(유실 없음).
